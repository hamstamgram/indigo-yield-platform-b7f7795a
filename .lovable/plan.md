

# Audit Log Cleanup — Purge 1.2 GB of Test-Data Artifacts

## The Problem

The database is carrying **1.23 GB** across two audit tables:
- `audit_log`: 623K rows / 774 MB
- `data_edit_audit`: 78K rows / 458 MB

**Root cause identified**: 614,022 of the 623K audit_log rows (98.5%) were generated during the week of March 30 — the test data purge. These logs record INSERT/DELETE/UPDATE of test transactions, test investor positions, and test yield distributions that no longer exist. They have zero diagnostic or compliance value.

Current weekly rate (excluding the purge week) is ~1,700 rows/week — completely healthy and sustainable.

## What the Migration Will Do

**Step 1 — Purge stale audit_log rows from the test-data cleanup period**
- Delete all `audit_log` rows where `created_at` between March 28 and April 5 AND the `action` is one of the bulk operations (`INSERT`, `DELETE`, `UPDATE`, `DELTA_UPDATE`) — preserving any `VOID`, `WITHDRAWAL_COMPLETE`, or other business-meaningful actions from that period.
- Expected deletion: ~610K rows, reclaiming ~760 MB.

**Step 2 — Purge corresponding data_edit_audit rows**
- Delete `data_edit_audit` rows from the same period that were generated by the test data wipe.
- Expected deletion: ~75K rows, reclaiming ~440 MB.

**Step 3 — VACUUM the freed space**
- Run `VACUUM` on both tables so Postgres actually returns the disk space.

**Step 4 — Add a retention policy function**
- Create `purge_old_audit_logs(retention_days int DEFAULT 90)` — a SECURITY DEFINER function that deletes audit records older than the retention period.
- This can be called manually or scheduled via pg_cron later.

## What We Preserve

- All VOID records (compliance-critical)
- All WITHDRAWAL_COMPLETE records  
- All FEE_SCHEDULE changes
- All INTEGRITY_ALERT records
- All audit records from April 6 onward (current operations)
- All records with business-meaningful actions regardless of date

## Risk Assessment

- **Risk**: None — we are deleting logs that recorded the creation and deletion of test data that itself was already deleted.
- **Reversibility**: The purged data is about test artifacts; PITR is available if needed.
- **Downtime**: None — DELETE + VACUUM runs online.

## Expected Result

| Table | Before | After |
|-------|--------|-------|
| `audit_log` | 774 MB (623K rows) | ~14 MB (~13K rows) |
| `data_edit_audit` | 458 MB (78K rows) | ~18 MB (~3K rows) |
| **Total** | **1.23 GB** | **~32 MB** |

## Technical Details

Single migration file with:
1. Batch DELETE on `audit_log` (WHERE created_at between cleanup dates AND action IN bulk ops)
2. Batch DELETE on `data_edit_audit` (WHERE edited_at between cleanup dates)  
3. `purge_old_audit_logs()` function for ongoing maintenance
4. Post-migration VACUUM via separate exec call (cannot run inside transaction)
