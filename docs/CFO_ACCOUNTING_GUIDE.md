**Version:** 2.0.0
**Date:** 2026-01-14
**Classification:** Internal - Finance Team
**Updated:** Aligned with comprehensive hardening v2

## Executive Summary

This document explains how the Indigo Yield Platform handles accounting for deposits, withdrawals, yield distributions, and fees. All calculations are performed in each fund's **native currency** (base asset) with no USD conversion in accounting tables.

---

## 1. Native Currency (Base Asset) Accounting

### Principle
Each fund operates in a single native currency, defined by the `funds.asset` field:

| Fund Code | Base Asset | Description |
|-----------|------------|-------------|
| ALPHA-01  | USDC       | All amounts in USDC |
| BETA-02   | ETH        | All amounts in ETH |
| GAMMA-03  | BTC        | All amounts in BTC |

### What This Means for Statements
- **All investor balances** are shown in the fund's base asset
- **All yield distributions** are calculated and recorded in base asset units
- **All fee calculations** are in base asset units
- **No USD valuations** appear in the core accounting ledger

### Example
```
Fund: ALPHA-01 (Base Asset: USDC)
Investor: John Smith
Position: 10,000.00 USDC
Yield Event: +150.00 USDC gross, -45.00 USDC fee, +105.00 USDC net
New Position: 10,105.00 USDC
```

## 2. Fee Model: Gross, Net, and Fees

### The Conservation Identity
For every yield event, the following identity **always** holds:

```
GROSS YIELD = NET YIELD + FEE AMOUNT
```

This is enforced at the database level and verified automatically.

### Definitions

| Term | Definition | Example |
|------|------------|---------|
| **Gross Yield** | Total yield earned before any fees | 150.00 USDC |
| **Fee %** | Performance fee percentage (investor-specific) | 30% |
| **Fee Amount** | Gross × Fee % | 45.00 USDC |
| **Net Yield** | What the investor keeps: Gross - Fee | 105.00 USDC |

### Fee Schedule
Each investor may have a custom fee percentage stored in `investor_fee_schedule`. If no custom schedule exists, the default is **30%**.

### Management Fee Policy (CFO Directive)
**IMPORTANT:** Per CFO policy, the platform has **NO management fees**:
- `mgmt_fee_bps = 0` is enforced at database level via CHECK constraint
- `mgmt_fees_paid = 0` is frozen on all investor positions
- The UI does not display any management fee information
- Only **performance fees** are charged (on yield distributions only)

This policy is enforced by database constraints and cannot be changed without schema migration.

## 4. Distribution Level Accounting

### How Distributions Aggregate

When the system distributes yield to all investors in a fund:

| Field | Description |
|-------|-------------|
| `gross_yield` | Total yield generated by the fund |
| `total_fees` | Sum of all investor fees |
| `net_yield` | Sum of all investor net yields |
| `dust_amount` | Rounding difference (allocated to largest investor) |

### Conservation at Distribution Level
```
gross_yield = SUM(all investor gross_yield_amount) + dust_amount
net_yield = SUM(all investor net_yield_amount)
total_fees = SUM(all investor fee_amount)

And: gross_yield ≈ net_yield + total_fees (within dust tolerance)
```

### Dust Handling
Due to rounding (10 decimal places), tiny differences may occur. These are:
- Tracked in the `dust_amount` field
- Allocated to the largest investor (by position size)
- Recorded with `dust_receiver_id`
- Enforced by DB constraint: |dust_amount| ≤ 0.01 base units per distribution

## 5. Transaction Purpose: Transaction vs. Reporting

### Two AUM Tracks
The system maintains two separate AUM values for different purposes:

| Purpose | Usage | Who Uses It |
|---------|-------|-------------|
| `transaction` | Actual deposits/withdrawals, yield crystallization | Operations team, investors |
| `reporting` | Month-end reporting, regulatory filings | Finance team, auditors |

### Why This Matters
- **Transaction AUM** drives actual money movement
- **Reporting AUM** may include adjustments for reporting periods
- They may differ slightly due to timing of month-end closes

### Database Enforcement
The system **rejects** any deposit or withdrawal that doesn't use `purpose='transaction'`. This prevents accidental mixing of accounting tracks.

## 6. Reconciliation Reports

### Daily Health Check
The system runs 11 automatic health checks that verify:

| Check | What It Validates | Severity |
|-------|-------------------|----------|
| YIELD_CONSERVATION | gross = net + fees for all distributions | CRITICAL |
| LEDGER_POSITION_MATCH | Positions match sum of deposits + yield | CRITICAL |
| NATIVE_CURRENCY | All transactions use fund's base asset | CRITICAL |
| NO_MANAGEMENT_FEE | mgmt_fee_bps = 0 on all funds | CRITICAL |
| EVENT_CONSERVATION | Yield events satisfy gross = net + fee | CRITICAL |
| ECONOMIC_DATE_NOT_NULL | All transactions have explicit tx_date | CRITICAL |
| AS_OF_FILTERING | No backdated entries violating period rules | WARNING |
| AUM_PURPOSE_CONSISTENCY | All AUM records have proper purpose | WARNING |
| DUPLICATE_PREFLOW_AUM | No duplicate preflow entries | WARNING |
| DUST_TOLERANCE | Dust amounts within asset-specific tolerance | WARNING |
| VOID_CASCADE_INTEGRITY | Voided transactions properly cascade | WARNING |

### Accessing Health Check Results
```sql
SELECT * FROM run_comprehensive_health_check();
```

All checks should show `check_status = 'PASS'` and `violation_count = 0`.

### If Violations Are Found
Contact the engineering team immediately. Violations indicate:
- Data integrity issues that need investigation
- Potential bugs in yield calculations
- Manual corrections that bypassed validation

## 7. Security Controls

### Who Can Execute Financial Operations

| Operation | Allowed Roles | Blocked Roles |
|-----------|---------------|---------------|
| Deposits | authenticated (admin) | PUBLIC, anon |
| Withdrawals | authenticated (admin) | PUBLIC, anon |
| View own position | authenticated (investor) | PUBLIC, anon |
| Run health checks | authenticated | PUBLIC, anon |

### Admin Verification
All mutating operations require `is_admin() = true` within the authenticated session.

## 8. Audit Trail

### What Gets Logged
Every deposit and withdrawal creates an `audit_log` entry containing:
- Action type (DEPOSIT_WITH_CRYSTALLIZATION, WITHDRAWAL_WITH_CRYSTALLIZATION)
- Before/after balances
- Crystallization details (yield allocated before the transaction)
- Administrator who executed the action
- Timestamp

### Accessing Audit Logs
```sql
SELECT * FROM audit_log
WHERE entity = 'transactions_v2'
ORDER BY created_at DESC;
```

## 9. Glossary

| Term | Definition |
|------|------------|
| **AUM** | Assets Under Management - total value held in a fund |
| **Base Asset** | The native currency of a fund (USDC, ETH, BTC, etc.) |
| **Crystallization** | Process of calculating and allocating accrued yield before a deposit/withdrawal |
| **Dust** | Small rounding differences (< 0.0001 units) |
| **Gross Yield** | Total yield before fees |
| **Net Yield** | Yield after fees (what investor receives) |
| **Performance Fee** | Percentage of yield taken as fee (default 30%) |
| **Purpose** | Transaction vs. Reporting track for AUM |
