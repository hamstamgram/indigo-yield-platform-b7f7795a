## Critical Findings

### 🔴 P0 - Data Corruption Risk

**282 RPCs have parameter order mismatches**

Example:
```typescript
// Database expects:
CREATE FUNCTION admin_create_transaction(
  p_investor_id uuid,      -- position 1
  p_fund_id uuid,          -- position 2
  p_type text,             -- position 3
  p_amount numeric,        -- position 4
  p_tx_date date           -- position 5
)

// Frontend contract has:
requiredParams: [
  "p_amount",      // position 1 (should be 4!)
  "p_fund_id",     // position 2 (correct)
  "p_investor_id", // position 3 (should be 1!)
  "p_tx_date",     // position 4 (should be 5!)
  "p_type"         // position 5 (should be 3!)
]

// Result when called:
// - Amount goes to investor_id field
// - Investor ID goes to type field
// - Type goes to amount field
// = DATA CORRUPTION
```

## Questions?

### Why are parameter orders wrong?
The frontend contract appears to have been auto-generated incorrectly or manually edited with alphabetical sorting. Supabase RPC calls are POSITIONAL - parameter order must match exactly.

### Is production broken right now?
Possibly. If any of the 282 mismatched RPCs are actively called, they're passing wrong data. Check application logs for RPC errors.

### How did this happen?
The contract was likely generated without respecting the `pg_proc.proargnames` order, or parameters were manually reordered for "readability" without realizing order matters.

### Can we fix this automatically?
Yes - recommended approach is to query `pg_proc` and regenerate the entire contract. See Option B in CRITICAL_RPCS.md.

### Will this affect users?
If the mismatched RPCs are used in production, yes. Financial transactions could have wrong amounts, wrong dates, or wrong investor attribution.

### How long to fix?
- Emergency hotfix (10 critical RPCs): 30 minutes
- Full regeneration: 2-3 hours
- Testing + deployment: 1-2 hours
- **Total**: 4-6 hours for complete fix
